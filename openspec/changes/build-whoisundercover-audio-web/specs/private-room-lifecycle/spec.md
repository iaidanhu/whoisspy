## ADDED Requirements

### Requirement: 私密房间码格式与唯一性

系统必须生成总长度为 8 的房间码：首字符为大写 `W`，后 7 位为随机字母数字字符。系统必须保证该码与当前所有活跃房间码不重复。

#### Scenario: 创建房间返回有效房间码

- **当** 房主创建新房间
- **则** 系统返回长度为 8、以 `W` 开头、且不与任何活跃房间码冲突的房间码

### Requirement: 无登录加入流程

系统必须允许玩家在未登录的情况下，通过房间码、昵称与头像标识加入私密房间。头像展示必须基于 **DiceBear Big Smile** 风格，客户端以 `avatarSeed`（字符串）与固定风格 `bigSmile` 生成头像；服务端保存 `displayName` 与 `avatarSeed`（或等价字段）供全房间同步展示。

#### Scenario: 无账号加入房间

- **当** 玩家提交房间码、昵称（`displayName`）与头像种子（`avatarSeed`）
- **则** 系统将其加入房间并分配会话内玩家身份，并在房间内展示对应昵称与 Big Smile 头像

### Requirement: 首页双态与选择页

客户端在 **`/[locale]/`** 必须实现两种互斥形态：**(A)** 无有效 **`whoisspy_profile_v1`** 时，仅展示契约登记（昵称、头像、指纹完成）；完成后写入 localStorage 并进入 **`/[locale]/start`**。**(B)** 已有有效档案时，**不得**将契约登记作为首屏；须 **直接进入** **`/[locale]/start`**（加入 / 创建选择），或 **`/[locale]/`** 在客户端等价渲染选择区（推荐前者：重定向至 **`/start`** 以保持路由清晰）。

在 **`/[locale]/start`**：用户选择 **加入房间** 时须输入房间号并确认后进入 **`/[locale]/r/[code]`**；选择 **创建房间** 时创建成功后 **立即** 进入 **`/[locale]/r/[code]`** 工作间。若访问 **`/[locale]/start`** 且无有效档案，须重定向 **`/[locale]/`**。站点根 **`/`** 重定向默认语言（如 **`/zh`**）。

#### Scenario: 已建档用户打开首页

- **当** 用户本地已有有效 `whoisspy_profile_v1` 并访问 **`/[locale]/`**
- **则** 客户端直接进入选择流程（重定向 **`/[locale]/start`** 或等价），不强制展示契约登记

#### Scenario: 首次用户完成登记

- **当** 用户首次完成契约登记并保存档案
- **则** 客户端进入 **`/[locale]/start`**

### Requirement: 创建后进工作间且房主为一号位

创建房间的用户必须为 **房主**，且在房间内座位序号为 **1**（加入顺序第一）。其它玩家按加入顺序递增座位号。

#### Scenario: 房主创建房间

- **当** 用户 A 创建房间成功
- **则** 用户 A 以房主身份进入 **`/[locale]/r/[code]`**，座位为 1 号

### Requirement: 房主专属控制与游戏设置默认值

在 **`/[locale]/r/[code]`** 工作间顶栏（或 `design-ui` 约定位置），**仅房主**可见 **游戏设置** 与 **开始游戏** 控件；**非房主**不得见上述控件。

**游戏设置**为可折叠面板或弹层，内包含至少：**投票阶段时长**、**发言阶段时长**、**会话总局数（1–10）** 等；**每一项均有默认值**，房主**不打开设置也可直接开局**（默认值生效）。**开始游戏** 在人数等条件不满足时（如少于 3 人）必须为 **禁用** 状态；满足条件后方可点击。

系统必须在房间或会话对象上持久化当前生效的配置（至少默认值可被读取）。

#### Scenario: 非房主看不到开始按钮

- **当** 普通玩家进入工作间
- **则** 其界面不展示「开始游戏」与「游戏设置」

#### Scenario: 人数不足开始禁用

- **当** 房间内仅有 2 名玩家且房主查看顶栏
- **则**「开始游戏」为禁用，直至第三人加入

### Requirement: 无准备阶段

系统不得实现独立的「准备」阶段或准备超时逻辑；玩家进入房间后即视为处于可开局状态，由房主控制开始游戏。

#### Scenario: 加入后即可被房主开局

- **当** 房间内玩家数不少于 3 且房主触发开始游戏
- **则** 系统进入发词与对局流程，不要求玩家额外点击准备

### Requirement: 房间人数与开局下限

系统必须将房间内玩家数限制在 3 至 8 人（含）。在玩家数少于 3 时，房主不得开始游戏。

#### Scenario: 房间已满拒绝加入

- **当** 第 9 名玩家尝试加入已达 8 人的房间
- **则** 系统拒绝加入并返回房间已满的提示

#### Scenario: 人数不足禁止开局

- **当** 房主在少于 3 名玩家时尝试开始游戏
- **则** 系统阻止开局并提示最少人数要求

### Requirement: 房主可配置会话局数与阶段时长

系统必须允许房主配置 **本会话总局数**（1～10）、**投票阶段默认时长**（默认 **30** 秒）、**发言阶段默认时长**（默认 **30** 秒）；不得提供房主自定义词组选项。

#### Scenario: 保存房主设置

- **当** 房主在游戏设置中修改参数并确认（或开局时采用当前配置）
- **则** 系统在对局中使用该配置驱动计时与局数上限

### Requirement: 仅使用全局系统词库

系统必须从全局系统词库中抽取词对发牌；不得允许房主或玩家提交自定义词组作为当局词语来源。词库数据以项目内 `data/system-word-pairs.json`（或经构建同步的等价数据源）为准。

#### Scenario: 开局发词来源为系统词库

- **当** 一局游戏开始需要分配词语
- **则** 系统仅从系统词库随机选取一对词语，并按规则分配给平民与唯一卧底
