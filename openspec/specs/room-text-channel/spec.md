## ADDED Requirements

### Requirement: 发言阶段仅当前发言人可发送描述文本（方案 A）

在处于「发言」阶段且轮到具体玩家发言时，系统必须仅允许该名玩家发送本轮**描述/发言**类文本消息；其他任何玩家发送同类消息的请求必须被拒绝，并返回可展示给用户的错误原因。

#### Scenario: 当前发言人发送描述

- **当** 游戏处于发言阶段且轮到座位号为 N 的玩家发言
- **则** 仅座位号为 N 的玩家发送的描述消息被接受并广播给房间内其他成员（或按产品约定的可见范围）

#### Scenario: 非当前发言人尝试发送描述

- **当** 游戏处于发言阶段但轮到另一名玩家发言
- **则** 系统拒绝该玩家的描述发送请求，且不将该文本写入房间消息流

### Requirement: 文字消息经服务端校验与广播

所有进入房间共享展示区的玩家文本消息必须经过服务端校验（身份、阶段、权限）后，再由服务端向房间内成员广播；客户端不得仅依赖本地乐观 UI 作为唯一真实来源。

#### Scenario: 广播合法描述消息

- **当** 服务端接受一条符合阶段与权限规则的描述消息
- **则** 服务端将该消息持久化或纳入当局事件流，并向房间内所有客户端推送该消息

### Requirement: 可选浏览器端语音转文字输入

客户端可提供基于浏览器 **Web Speech API**（或等价能力）的**可选**输入方式：在用户明确操作（如点击 **中部 Textarea 旁的麦克风图标** 或「语音输入」）时进行识别，将识别结果 **填入同一多行输入**，用户可编辑后发送。当 API 不可用、用户拒绝麦克风权限或识别失败时，客户端必须降级为仅键盘输入，且不得阻塞对局进行。

#### Scenario: STT 成功并发送

- **当** 用户在轮到本人发言时使用语音输入且浏览器返回识别文本
- **则** 客户端将文本填入 Textarea，用户确认发送后，服务端按与普通文本相同的规则校验并广播

#### Scenario: STT 不可用时的降级

- **当** 浏览器不支持语音识别或用户拒绝麦克风权限
- **则** 客户端隐藏或禁用语音输入入口，用户仍可通过键盘完成发言

### Requirement: 发言阶段主舞台与侧栏联动（对齐 UI 参考图）

在 **发言阶段**，客户端 UI 必须与 **`design-ui.md` §8.2～§8.3** 一致，并与仓库参考 **[`docs/images/image.png`](../../../../../docs/images/image.png)** 同构：

- **侧栏**：**当前发言人** 玩家卡片须 **高亮**（金色描边/发光、可略提亮底），使全员知晓「轮到谁」。
- **中部舞台**：
  - **当当前发言人不是本机**：展示该玩家的 **大号头像** + **其已广播的描述文本**（宽条主文案区）；未发送前显示占位提示（如请 N 号发言）。
  - **当当前发言人是本机**：中部同一区域切换为 **输入态**：**多行 Textarea**（**Enter 发送、Shift+Enter 换行**，见 `design-ui.md` §8.3）+ **麦克风（语音转文字）**；提交成功后 **结束本轮本人发言**（清空本地输入、等待状态机推进），**不得**与「离开房间」混淆。

#### Scenario: 他人发言时全员见舞台

- **当** 轮到座位 N 发言且 N 不是本地用户
- **则** 所有客户端侧栏高亮 N 号卡片，中部展示 N 的大头像与其描述内容（或占位提示）

#### Scenario: 本人发言输入与发送

- **当** 轮到本地用户发言
- **则** 侧栏高亮本人卡片，中部展示 Textarea 与麦克风入口；用户按 **Enter** 发送、**Shift+Enter** 换行；发送后客户端提交文本并退出输入态

### Requirement: 等待开局背景音乐循环

在 **`/[locale]/r/[code]`** 工作间内，当房间处于 **等待房主开始游戏**（本会话尚未进入任意一局的进行中状态，玩家在房内等待）时，客户端必须 **循环播放** 背景音乐以提升沉浸感。

- **音源**：以仓库内 **[`docs/audio/day.mp3`](../../../../../docs/audio/day.mp3)** 为权威源；构建后须通过 **`public/` 或等价静态路径** 提供可请求的同一音频（实现路径可不同，但须文档化）。
- **行为**：使用 `HTMLAudioElement`（或等价）**`loop = true`**；进入 **对局进行中**（发词后的发言/投票等任一阶段）时 **停止或淡出 BGM**；本会话 **终局榜** 展示期间 **不强制** 播放 BGM（默认停止或保持静音直至回到等待态）。房主触发 **重新开始游戏** 回到等待态后，在用户未关闭声音的前提下 **恢复** 循环播放。
- **用户控制**：须与顶栏声音设置联动（见 `design-ui.md`）：关闭 BGM 或总静音后不得自动以最大音量恢复，直至用户重新开启。

#### Scenario: 成员进房等待房主开局

- **当** 玩家已进入房间且会话仍处于等待开局状态
- **则** 客户端在声音开启且浏览器策略允许时播放 `day.mp3` 并循环

#### Scenario: 房主点击开始游戏

- **当** 房主成功触发开局，状态机进入对局进行中
- **则** 客户端停止或淡出等待 BGM

### Requirement: 开局、局终与临超时短音效（资源待定）

除已有发言提示与投票音效外，产品规划在以下时机增加 **短音效**；**音频文件尚未提供时**，实现侧可保留事件钩子与静音占位，**不得阻塞**对局流程。

- **游戏开始**：从等待开局切入 **第一局或新一局** 进行中时播放一次（资源待定）。
- **临近超时**：发言或投票倒计时进入 **预警区间** 时播放预警类音效（可与现有「倒计时/预警」合并或分层；资源待定）。
- **单局结束**：当局胜负已定时播放结算类音效（资源待定）。
- **本会话结束**：全部局数完成、进入终局排行榜展示时播放结束类音效（可与单局音效用同一文件或分文件；资源待定）。

#### Scenario: 资源未就绪

- **当** 上述某类音效文件尚未放入仓库
- **则** 客户端跳过该次播放且不报错；事件仍按状态机正常推进

### Requirement: 发言轮次提示音

系统必须在新的发言轮次开始时播放可听提示，内容包括玩家序号提示（例如「请 2 号发言」）。

#### Scenario: 切换到下一位发言者

- **当** 发言阶段推进到下一名玩家
- **则** 系统在轮次开始当时或之前播放对应发言提示音

### Requirement: 投票与超时音效

系统必须在投票操作以及**发言**、**投票**阶段的超时预警时提供音效反馈（本游戏无准备阶段，不包含准备阶段音效要求）。

#### Scenario: 投票提交确认

- **当** 玩家提交投票
- **则** 系统在该玩家客户端播放投票确认音效

#### Scenario: 发言剩余时间进入预警区间

- **当** 当前发言剩余时间进入配置的预警阈值
- **则** 系统播放倒计时/预警类音效

### Requirement: 消息长度与频率限制

系统必须对玩家文本消息实施合理的长度上限与发送频率限制，以降低刷屏与滥用风险。

#### Scenario: 超出长度上限

- **当** 玩家提交超过配置最大长度的文本
- **则** 服务端拒绝该消息并提示长度限制
